# N1/N2/N3 지수 규명 (2026-06-18)

## 결론
adlog/adpin의 **N1(유사도)·N2(관련성)·N3(랭킹)는 순위·검색량·업체수로 계산할 수 없고, 네이버도 노출하지 않는 (매장,키워드)별 고유 적합도 점수**다. 따라서 와벨리에서 동일 수치를 산출/표시하는 것은 불가능. 잘못된 순위기반 N(예전 nscore)을 제거하고, **정확히 맞는 실측 지표(실제 순위·노출수·최고/평균순위·저장·리뷰·노출점수)**로 교정했다.

## 근거 (adpin 실측 9행 분석 — `260616/.../nformula.py`)
사장님이 제공한 adpin 9건(모토모토 7 + 와우가우가그릴 2)을 분석:

1. **N3는 실시간 순위의 함수가 아님.** 모토모토 "춘천 동면카페"=**8위**와 "춘천 카페"=**141위**가 **N1·N3 모두 6자리까지 동일**(N1 0.397908 / N3 0.368760). N3↔순위 상관계수 ≈ +0.02. → 우리 nscore(순위→N 회귀)가 안 맞던 근본 이유.
2. **N2는 매장 고정값.** 모토모토 비브랜드 6개 키워드 N2 전부 `0.390296`, 와우가우가그릴 `0.368564`. 매장 콘텐츠(리뷰 등) 기반, 키워드 무관(브랜드검색만 미세하게 낮음).
3. **N1·N3는 상관 0.999** — 같은 '키워드↔매장 관련성'에서 파생. 검색량·업체수와 무관(같은 N1인데 업체수 2269 vs 878).
4. **텍스트/숫자 피처로 설명 불가.** "춘천 카페"≡"춘천 동면카페"(N 정확히 동일)인데 "춘천 대형카페"는 다름 → 표면 텍스트 규칙 없음(임베딩류 추정).

## 출처 추적 (전부 막힘 확인)
- **네이버 검색응답에 적합도 필드 없음** — `probe_relevance.py`로 모토모토 노드 전체 덤프, 0.397908 등 어디에도 없음. 유일한 점수는 리뷰평점(visitorReviewScore 4.36).
- **adlog 클라이언트 JS에도 없음** — 대시보드가 데이터를 서버 렌더링(별도 데이터 XHR 없음), common.js는 일반 게시판 프레임워크. → adlog 서버가 자체 계산.
- 즉 N값의 원천 = adlog/adpin 자체 모델(서버). 읽어오려면 그 서비스를 스크래핑하는 수밖에 없음(= adlog를 그대로 미러링, 취약).

## 추정식 (2026-06-18, adpin 9행 회귀 — `nestimate.py`)
정확공식은 불가하지만 **추정**은 가능. 9행 회귀로 도출:
```
N1(유사도) ≈ 0.70958 − 0.11139·log₁₀(전체업체수) + 0.0205·log₁₀(검색량)   R²≈0.75, 오차 ±0.03(최대 ±0.08)
N3(랭킹)  ≈ 0.25756 + 0.27668·N1                                        R²≈0.998
N2(관련성) ≈ 0.35267 + 0.01124·log₁₀(방문리뷰)                          [매장 단위]
```
핵심: N1은 경쟁강도(업체수)↓·검색량↑. 업체수<30(브랜드/단독)은 추정 제외(빗나감). 종합 = 노출 키워드별 추정 N의 평균.

## 와벨리 적용 (옵션 A — 추정 N 표시)
- 수집기: 키워드별 **전체업체수**를 캡처(`naver_browser._cur_total` → discover/index → place_rankings.total_biz).
- 워커(`wavely_sync._analyze_store` + `wavely_index.push`): `nestimate.store_indices`로 추정 N1/N2/N3 산출·적재(온디맨드·일일 동일).
- 앱(`place-rank.tsx`): 매장 종합 **"N지수(추정치·참고용)"** 박스 + 키워드별 **N1** 표시. "추정치, 타 서비스와 ±0.05 차이" 안내.
- **순위·저장·리뷰는 실측**(adpin과 일치 검증). N만 추정.
