# 플레이스랭크(place_collector) → 와벨리 사장님 기능 연동 검토

> 결론 먼저: **수집기(스크래퍼) 자체는 와벨리에 못 넣는다. 하지만 "수집 결과"를 와벨리 안 사장님 전용 대시보드로 보여주는 건 가능하고, 오히려 정석 구조다.**

## ❌ 왜 스크래퍼 자체는 못 넣나
플레이스랭크 수집기는 **Python + Playwright(진짜 크롬) + 가정용 IP**가 필수다. (수집기의 `DEPLOY.md`에 명시: 네이버는 데이터센터/클라우드 IP를 봇으로 보고 차단 → 클라우드에서 돌리면 0건)

와벨리의 실행 환경은 전부 그 조건과 안 맞음:
- **Expo 앱(클라이언트)**: 사용자 폰/브라우저 — Playwright 못 돌림, 네이버 토큰/쿠키 노출 위험.
- **Supabase Edge Functions(Deno)**: 서버리스 + 데이터센터 IP → 네이버 차단 + 브라우저 없음 + 실행시간 제한.
- **Vercel**: 정적 호스팅 — 수집 불가.

→ 수집은 지금처럼 **집 PC(가정 IP)에서** 계속 돌려야 한다. 이건 제약이지 설계 실수가 아님.

## ✅ 가능한 구조 (수집기의 기존 "분리 설계"와 정확히 일치)
플레이스랭크는 이미 **수집기 / DB / 웹UI 분리** 구조이고, DEPLOY.md도 "DB를 공용(Postgres)으로 승격" 경로를 언급함. 그 공용 DB를 **Supabase**로 두면 끝.

```
[집 PC 수집기]  Playwright (지금 그대로)
     │  매일 수집 → 결과를 Supabase 테이블에 push (service key, ~30줄 추가)
     ▼
[Supabase]  place_rankings 테이블  ── RLS: 매장 소유자만 본인 데이터 조회
     ▲
[와벨리 앱]  사장님이 본인 매장 ↔ 네이버 place_id 연결 → 순위 추이 대시보드
```

핵심: 와벨리는 **읽기만** 한다(수집 안 함). 네이버를 건드리는 부분은 전부 집 PC에 남는다.

## 와벨리 쪽에서 내가 바로 만들 수 있는 것
- `place_rankings` 테이블 + **RLS**(매장 소유자·관리자만 SELECT — 남의 순위 못 봄)
- 매장↔네이버 연결: **`stores.naver_place_id` 컬럼이 이미 있음** → 사장님 매장관리 화면에 place_id 입력칸만 추가
- **사장님 전용 순위 대시보드** 화면(키워드별 순위 추이·저장수·리뷰수). `biz_verified` 매장 소유자에게만 노출
- (선택) 빠른메뉴/매장관리에서 진입

## 수집기 쪽 변경(작음)
- `scheduler/daily.py` 끝에 "수집결과를 Supabase REST로 upsert" 하는 함수 추가(약 30줄). 기존 SQLite 적재는 그대로 두고 **미러링**만.

## 🔴 먼저 정해야 할 제품 결정
1. **매장↔place_id 연결 방식**: 사장님이 직접 입력? 관리자 승인 후 활성화? (악용방지: 남의 매장 place_id 넣는 것 막기 — 본인 인증매장만 허용 권장)
2. **추적 키워드**: 사장님이 지정? 업종 기반 자동 추천? (수집 부하·RATE 제한과 직결 — 매장당 N개 제한 필요)
3. **무료 기본 vs 유료 부가기능**: 사장님 대상 핵심 유료기능 후보. 광고지갑과 묶을지.
4. **수집 빈도/대상 범위**: 지금 집PC 하루 2회. 사장님이 늘면 RATE(2.5~5초, 일일3000) 안에서 매장수 한계 계산 필요.
5. **사장님 자격**: `biz_verified` 통과 매장 소유자만? (신뢰필드라 서버에서만 부여 — 이미 그렇게 하드닝됨)

## 권장 진행안 (MVP)
1) 와벨리: `place_rankings` 테이블+RLS, 매장관리에 place_id 연결칸, 사장님 순위 대시보드(1매장·키워드 소수) — **내가 작업 가능**
2) 수집기: daily.py에 Supabase push 추가 — **내가 작업 가능**(수집기 폴더 접근됨)
3) 단, 위 1~2번 제품결정(특히 연결방식·키워드정책·유무료)을 먼저 확정해야 표/화면을 안 헛 만든다.
