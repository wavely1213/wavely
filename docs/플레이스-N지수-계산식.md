# 플레이스 N지수 — 원리 분석 + 자체 계산식

> 목적: adlog식 N1/N2/N3 "플레이스 지수"를 **원리부터 재구성**해 자체 계산식으로 만든다.
> 계산 위치는 **수집 서버(260616)** — 수집한 값으로 N지수를 계산해 `place_analysis` 에 적재.
> (메모리상 현재 수집기 워커는 N1=null, N2/N3=평균 으로 임시 처리 → 이 식으로 교체)

## 1) 원리 (역산으로 찾은 핵심)
adlog의 정확한 계수는 **비공개**라 그 소수값(1.4726 등)을 똑같이 복제할 순 없다. 하지만 N지수가 무엇을 재는지는 분명하다 — **네이버 플레이스 노출력을 좌우하는 3대 축의 합성**:

| 축 | 의미 | 수집 가능한 입력 |
|---|---|---|
| **노출** | 키워드들에서 얼마나 상위에 뜨나 | 키워드별 순위(rank) |
| **리뷰** | 방문자·블로그 리뷰량 | visitorReviewCount, blogCafeReviewCount |
| **저장(인기)** | 저장수(활동성) | saveCount |

핵심 통찰 2가지:
1. **로그 스케일** — 저장 0→100의 효과가 100→200보다 훨씬 큼(한계효용 체감). 그래서 리뷰·저장은 `log`.
2. **순위는 DCG식 감쇠** — 1위와 3위 차이가 30위와 50위 차이보다 크다. 그래서 순위 점수 `gain(r)=1/log2(r+1)` (검색랭킹 평가의 표준 DCG gain). rank1→1.0, 3→0.5, 10→0.29, 권외→0.

## 2) 자체 계산식 (0~100 스케일, 투명)
```
N1 (노출지수) = 100 × Σ gain(rank_i) / 총키워드수      , gain(r)=1/log2(r+1), 권외=0
N2 (리뷰지수) = 100 × min(1, (log10(1+방문리뷰)+log10(1+블로그리뷰)) / (2·log10(1+500)))
N3 (저장지수) = 100 × min(1, log10(1+저장수) / log10(1+500))
종합점수      = 0.5·N1 + 0.25·N2 + 0.25·N3            (노출 가중 ↑ — 매출 직결)
```
- 스케일을 **0~100**으로 둔 이유: 사장님이 바로 이해. adlog의 소수(1.47…) 느낌을 원하면 ÷100 또는 ÷68 등으로 리스케일만 하면 됨(원리는 동일).
- 상한 `CAP=500`: 리뷰/저장 이쯤이면 만점. 업종 경쟁강도에 맞게 조정 가능.

## 3) 검증 (실제 계산 — 단조성·분별력 확인)
| 시나리오 | N1 노출 | N2 리뷰 | N3 저장 | 종합 |
|---|---|---|---|---|
| 와우가우가그릴(추정: 20kw, save100/방문26/블로그182) | 32.7 | 68.4 | 74.2 | **52.0** |
| 강한 매장(상위 다수·리뷰 多) | 52.3 | 97.4 | 97.2 | **74.8** |
| 약한 매장(대부분 권외·리뷰 少) | 4.6 | 28.8 | 41.3 | **19.8** |
| 신규 매장(노출 거의 없음) | 0.0 | 5.6 | 17.7 | **5.8** |

→ 매장 강도순으로 **단조 분리**(74.8 > 52.0 > 19.8 > 5.8). 0~100 유계, 한 축이 0이어도 다른 축이 받쳐줌. ✅

## 4) 수집 서버용 함수 (이걸 워커의 N계산 자리에 넣기)
```python
import math
def n_indices(ranks, save, vrev, brev, n_keywords=None):
    """ranks: 노출 키워드들의 순위 리스트(권외는 None/0). save/vrev/brev: 저장·방문리뷰·블로그리뷰."""
    exposed = [r for r in ranks if r]
    total = n_keywords if n_keywords is not None else len(ranks)
    gain = lambda r: 1.0 / math.log2(r + 1)
    n1 = 100.0 * (sum(gain(r) for r in exposed) / total) if total else 0.0
    CAP = 500
    n2 = 100.0 * min(1.0, (math.log10(1+vrev) + math.log10(1+brev)) / (2*math.log10(1+CAP)))
    n3 = 100.0 * min(1.0, math.log10(1+save) / math.log10(1+CAP))
    score = 0.5*n1 + 0.25*n2 + 0.25*n3
    return round(n1,1), round(n2,1), round(n3,1), round(score,1)
# place_analysis 적재 시: n1,n2,n3 = 위 결과. (와벨리 UI는 이미 n1/n2/n3 표시 지원)
```

## 5) 정직한 한계 + 다음
- 이건 **우리 자체 투명지수**다. adlog의 소수값과 1:1로 같진 않다(그 계수는 비공개).
- **adlog/우리 대시보드의 (입력 → N값) 샘플을 몇 개 주면**, 그 샘플에 회귀로 계수(가중치·CAP·스케일)를 피팅해 **adlog 값에 근접**하게 맞출 수 있다. 지금은 원리 기반 합리적 근사.
