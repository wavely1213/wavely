# 플레이스 N지수 — 원리 (수집기 역산 결과 기준)

> ⚠️ 갱신(2026-06-17): 수집기 `collector/nscore.py`에 **adlog 300건 회귀분석으로 도출한 실제 공식**이 이미 있다.
> 내가 앞서 만든 3요소(저장·리뷰·노출) 자작식은 **원리가 틀렸다(폐기)**. 진짜는 아래 — 순위(rank) 기반.

## 1) 수집기가 역산한 진짜 원리 (nscore.py)
adlog 데이터 300건 회귀분석 결과:
- **N1 = 키워드 고정 점수** (순위 무관, 키워드/카테고리로 결정). → 공식 없음, 현재 미산출(null).
- **N2 = 순위 경쟁력(증폭형)**: `N2 = 0.533802 − 0.00445882 · rank^0.58`   (R²=0.989)
- **N3 = 순위 경쟁력(압축형)**: `N3 = 0.368526 − 0.00038760 · rank^0.56`   (R²=0.996)
- N2·N3 모두 `rank^p` 멱함수. 순위 1위→최고, 순위 내려갈수록 감소. N3→순위 역산 ±5위.
- 키워드/지역마다 절편(C)이 달라질 수 있어 `nscore.calibrate()`로 키워드별 실측 2~3점 보정 권장.

자가검증(nscore.py): 1위 0.368068, 10위 0.367078, 50위 0.365236, 100위 0.363493, 300위 0.359342 — 예측 오차 미미.

## 2) 수집기 워커가 place_analysis 에 적재하는 값 (wavely_sync.py)
- `n1 = null` (공식 없음)
- `n2 = mean(predict_n2(rank))` — 노출 키워드들의 평균
- `n3 = mean(predict_n3(rank))` — 〃
- save_count / visitor_review / blog_review / keyword_count / exposed_count / best_rank / avg_rank
- ⚠️ **등록 키워드(place_rank_keywords)만 분석. 자동 키워드 탐색 미지원.** → 와벨리 UI도 키워드 등록식으로 정렬함(2026-06-17 수정).

## 3) 와벨리 화면이 보여주는 것
- **노출 점수(0~100)** = 와벨리가 client에서 키워드 순위로 계산(DCG): `100 · Σ(1/log2(rank+1)) / 총키워드`. N지수와 별개의 직관적 종합지표(사장님용 headline). rank1→100, 3→50, 10→29, 권외→0.
- **N2 / N3 지수** = 수집기가 적재한 adlog식 값(소수). N1은 null이라 미표시.
- 요약(총/TOP3/TOP10/상승/하락), 급등·급락, 순위분포, 키워드별 순위·변화 — 모두 place_rankings로 client 계산.
